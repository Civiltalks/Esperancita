Por que isso separa amador de produção?

"Agents are 30% of the work. The other 70% is the immune system." — Eric Siu

Agentes quebram silenciosamente. Crons falham sem avisar. Sub-agents travam no limbo. Sem monitoramento, você descobre problemas dias depois — quando já deu merda.

😰 Sem immune system

  • Cron falhou e ninguém viu
  • Sub-agent no limbo há 3 dias
  • Custo explotou sem saber por quê
  • Secrets expostos em arquivo .md
  • Backup? Qual backup?

🛡️ Com immune system

  • Watchdog detecta falha em 1h
  • sessions_yield garante resultado
  • Cost tracking em tempo real
  • openclaw secrets audit + openclaw doctor
  • Backup antes de cada mudança

1. Watchdog de Crons

Crie um cron que monitora os outros crons. Se algum falhar, retry automático até 3x. Se falhar 3x, alerta no Telegram.

Cron — Monitor de Crons (Diário 08h BRT)
{ "name": "Watchdog - Monitor de Crons", "schedule": { "kind": "cron", "expr": "0 8 * * *", "tz": "America/Sao_Paulo" }, "sessionTarget": "isolated", "payload": { "kind": "agentTurn", "message": "Checar saúde de todos os crons. Listar os que falharam nas últimas 24h. Fazer retry dos que falharam. Se algum falhar 3x, alertar no Telegram." }, "delivery": { "mode": "announce" } }

Lógica do watchdog

  1. Listar todos os crons ativos
  2. Checar último run de cada um
  3. Se falhou → retry automático (até 3x)
  4. Se falhou 3x → alertar no Telegram

⚠️ Importante: O watchdog também precisa de watchdog? Sim — configure um watchdog secundário que roda em horário diferente (ex: 20h BRT) para cobrir falhas no ciclo matinal.

2. Feedback Loops

Sistema de aprendizado contínuo: o agente aprende com suas decisões (approve/reject).

Estrutura de arquivos

memory/feedback/
├── content.json         ← feedback sobre conteúdo
├── tasks.json           ← feedback sobre tasks
└── recommendations.json ← feedback sobre sugestões

Formato de cada entrada

{
  "entries": [{
    "date": "2026-04-11",
    "context": "Sugeri thread sobre X",
    "decision": "reject",
    "reason": "Tom inadequado",
    "tags": ["linkedin", "thread"]
  }]
}
// Max 30 entradas por arquivo (FIFO)
Ciclo Feedback (JSON granular)
→
Mensal Lessons (curado, prose)
→
Permanente Decisions (política)

Regra: Agente DEVE consultar feedback antes de sugerir novamente → evita repetir erros. Isso é aprendizado real, não decorativo.

3. Monitoramento de Custos

Uso Modelo Custo relativo Exemplo mensal
Interação direta GPT-4o $$$ ~$0.50–1.00/sessão
Crons e automação GPT-4o-mini $ ~$0.01–0.05/sessão
Heartbeats GPT-4o-mini ¢ ~$1-2/mês

Regras de ouro

  • TODOS os crons devem rodar em GPT-4o-mini (nunca GPT-4o)
  • Heartbeats em GPT-4o-mini
  • Só a interação direta usa GPT-4o

Kill switch

  • 2x custo esperado → ⚠️ Alerta no Telegram
  • 3x custo esperado → 🚨 Pausa e pergunta a você

⚠️ NUNCA auto-kill. Sempre pedir confirmação humana antes de parar qualquer coisa.

4. Sub-agents + sessions_yield (v2026.3.13+)

Todo sub-agent spawnado DEVE ter follow-up. Use sessions_yield para encerrar o turno limpo após spawnar.

1 Spawna com sessions_spawn
→
2 Informa o que vai fazer
→
3 sessions_yield encerra turno
→
4 Resultado na próxima mensagem

✅ Sucesso

Resumir resultado em linguagem humana simples

❌ Falha

Retry imediato → se falhar 2x → avisar o usuário

🚫 Limbo

NUNCA deixar sub-agent sem follow-up — é o erro mais comum

5. Backup antes de mudanças

Regra

Antes de criar agentes, modificar config, ou reorganizar workspace:

mkdir -p backups/$(date +%Y-%m-%d)
cp ~/.openclaw/openclaw.json backups/$(date +%Y-%m-%d)/

ROLLBACK.md

Para cada mudança estrutural, crie um ROLLBACK.md junto:

## Como reverter
1. Copiar backup de /backups/YYYY-MM-DD/
2. Substituir arquivo em ~/.openclaw/
3. openclaw gateway restart

6. openclaw secrets audit (v2026.3.2+)

⚠️ Execute isso AGORA se nunca fez. Credenciais expostas são o ataque mais comum contra agentes mal configurados.

# Auditar todo o workspace
openclaw secrets audit

# Auditar diretório específico
openclaw secrets audit --path /root/.openclaw/workspace-meu-agente

# Relatório detalhado
openclaw secrets audit --report
O que detecta Arquivos alcançados
API keys hardcodadas.json, .md, .env
Tokens no git historyTodos os arquivos versionados
Credenciais em SOUL/AGENTS/TOOLSArquivos de agente
Patterns conhecidosOpenAI, Stripe, Telegram, AWS, etc.

exec.approvals — Nunca desabilite (v2026.3.13)

🔴 CRÍTICO: Em março/2026, 7 formas de burlar o sistema de approvals foram encontradas e corrigidas. Atacantes usavam caracteres invisíveis Unicode, quebras de linha com backslash, e wrappers de ferramentas comuns (pnpm, npm, Perl). O sistema existe para bloquear isso.

Verificar configuração

openclaw config get exec.approvals
# Deve retornar: ask

⚠️ Valores

  • ask ✅ — Pausa e pede confirmação
  • allow 🚨 — Executa SEM confirmar (NUNCA use)
  • deny — Bloqueia tudo

Por que não desabilitar?

Quando o agente quer executar algo fora do padrão, ele pausa e pede confirmação antes de prosseguir. É sua última linha de defesa.

Dica: Peça ao aluno para demonstrar ao vivo o sistema pausando e pedindo aprovação. O aluno tende a achar que é burocracia — mostrar o contexto de segurança muda completamente a percepção.
